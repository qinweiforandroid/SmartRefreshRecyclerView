# AI 改造项目起点梳理

> 问题文件类型：按具体问题单独记录。
> 建议后续同类问题继续新建独立文件，而不是追加到总表中。

### 现象

- 项目是多模块 Android RecyclerView 能力库。
- 已存在可用功能，但结构和文档仍有可整理空间。

### 初步观察

- `SmartRecyclerView` 与 `SwipeRecyclerView` 的 load more 滚动判断逻辑高度相似。
- `SmartV2RecyclerView` 表示项目内部可能已经在尝试新的实现方向。
- `SmartListCompat` 在易用性上有价值，但职责偏重。
- README 能用，但还不够匹配当前工程的真实结构和推荐姿势。

### 问题拆解

1. 是否应该先做 API 收敛？
2. 是否应该先做公共逻辑抽取？
3. 文档和代码哪个优先？

### 当前判断

- 优先做公共逻辑抽取更稳妥。
- 在未补文档前，继续大改会提高沟通成本。
- 在未补测试前，不适合做过深结构调整。

### 决策

- 先建立文档目录，沉淀设计和问题思考。
- 第一轮代码改造聚焦“低风险、高收益”的重复逻辑收敛。

### 待确认

- `SmartRecyclerView` 和 `SmartV2RecyclerView` 是否都要长期保留。
- sample 的推荐展示路径是否需要重构。
- 后续是否引入更现代的状态表达方式，例如 `Flow` 风格接口。

### 下一步

- 细化 PR1 设计
- 明确公共抽象边界
- 再进入代码改造阶段
